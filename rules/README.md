# Rules index

Эта папка содержит рабочие правила модернизации librw. Они нужны не как
декларация вкуса, а как практический способ безопасно переводить старый
RenderWare-style C/C++ код в современный C++.

## Порядок чтения

1. `../AGENTS.md` - общий контракт для разработки в репозитории.
2. `modern-cpp.md` - базовые правила языка и API.
3. `standard-library.md` - карта стандартной библиотеки: что использовать,
   где использовать и где быть осторожным.
4. `legacy-porting.md` - рецепты миграции старого кода.
5. `engine-backends.md` - правила для рендера, GPU/backend-кода, оконных
   библиотек и ImGui.

## Быстрый чеклист модернизации

- Начинай с поведения: что участок кода делает для формата, движка,
  backend-а или tool-а.
- Опиши ownership: value, unique owner, shared owner, borrowed view,
  platform handle.
- Замени C arrays на `std::array`, `std::vector`, `std::span`, `std::mdspan`.
- Замени plain `enum` на `enum class`, а bit flags - на typed flags.
- Замени raw pointers на references, `std::span`, `std::unique_ptr`,
  `std::shared_ptr` или RAII-handle, в зависимости от lifetime.
- Замени макросы на `constexpr`, `inline`, templates или attributes.
- Замени ручные циклы на algorithms/ranges, если это не ухудшает hot path.
- Замени `bool + out parameter` на `std::expected`, если нужно вернуть ошибку.
- Для временных и frame allocations рассматривай `std::pmr`.
- Для D3D/GL/SDL ресурсов всегда нужен RAII или явная cleanup-стратегия,
  которая работает при early return.

## Лестница решений

Выбирай решение сверху вниз:

1. Стандартная библиотека C++.
2. Уже существующая инфраструктура librw.
3. Небольшой typed wrapper вокруг платформенного API.
4. Новая зависимость только после архитектурной причины.
5. Самодельная структура данных или allocator только после профилирования
   или доказанной невозможности использовать готовое решение.

## Что считать хорошим результатом

Хороший современный участок librw:

- явно выражает размер и lifetime данных;
- не требует читать весь файл, чтобы понять кто освобождает ресурс;
- сохраняет binary/file compatibility;
- компилируется на MSVC/Clang/GCC в целевых профилях;
- отделяет platform-independent данные от D3D/GL/SDL деталей;
- содержит меньше макросов и меньше неявных integer flags;
- допускает будущую оптимизацию без изменения публичного API.
